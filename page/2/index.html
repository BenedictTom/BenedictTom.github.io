享元模式享元模式（Flyweight Pattern）是一种结构型设计模式，旨在最小化内存使用量或减少对象创建的数量，通过共享尽可能多的数据来实现。它特别适用于处理大量细粒度的对象，这些对象之间存在大量的重复数据，而这些重复数据可以被多个对象共享。 核心思想享元模式的核心在于区分对象的内部状态和外部状态： 内部状态：可以在多个对象实例间共享的状态，通常是不变的。 外部状态：随环境变化而变化的状态，不能被共享，通常由客户端在需要时传递给享元对象。 通过这种方式，享元模式能够显著减少应用程序中对象的数量，从而降低内存消耗，并可能提高性能。 享元模式的主要组件 Flyweight（抽象享元角色）：定义了一个接口，所有具体享元类都必须实现此接口。这个角色是可选的，但有助于实现更灵活的设计。 ConcreteFlyweight（具体享元角色）：实现了Flyweight接口，包含了可以共享的部分。需要注意的是，具体享元角色不能修改其内部状态。 UnsharedConcreteFlyweight（不可共享的具体享元角色）：并非所有的享元对象都可以被共享，某些情况下，某些对象可能包含无法共享的 ...

1. BIO (Blocking I/O) - 同步阻塞 I/O 核心思想：传统的 Java I/O 模型，基于流（Stream）进行数据传输。当一个线程发起 I/O 操作（如读取文件、网络数据）时，该线程会被阻塞，直到数据准备就绪并完成传输后，线程才能继续执行后续代码。 工作模式： 服务器端通常为每个客户端连接创建一个独立的线程来处理。 这个线程负责接收数据、处理请求、发送响应。 在等待 I/O 操作（如 socket.read()）完成期间，该线程处于阻塞状态，无法处理其他任务。 优点： 编程模型简单直观：逻辑清晰，易于理解和实现。 符合传统编程习惯：代码顺序执行，调试相对容易。 缺点： 资源消耗巨大：每个连接都需要一个独立线程。在高并发场景下（成千上万个连接），会创建大量线程，导致系统资源（CPU、内存）被大量消耗，线程上下文切换开销巨大。 扩展性差：受限于线程数量，无法有效支持海量并发连接。 线程利用率低：大部分线程在大部分时间里都在阻塞等待 I/O，真正执行业务逻辑的时间占比很低。 适用场景：连接 ...
