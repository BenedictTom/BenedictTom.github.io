1. 自我介绍项目介绍

回答要点：

  • 个人背景：学历、工作年限、技术栈
  • 项目背景：业务场景、技术选型、团队规模
  • 个人职责：负责的模块、解决的问题
  • 技术亮点：性能优化、架构设计、难点攻克

示例：
“我是XX，有X年Java开发经验，主要技术栈包括Spring Boot、Redis、RocketMQ、Elasticsearch等。目前负责电商平台的订单和支付系统，团队规模X人。我主要负责订单流程设计、支付集成、性能优化等工作。在项目中解决了高并发订单处理、分布式事务、搜索性能优化等技术难点。”

2. Redis为什么这么快

核心原因：

  1. 内存存储：数据存储在内存中，读写速度比磁盘快几个数量级
  2. 单线程模型：避免了多线程竞争和锁开销
  3. 高效的数据结构：专门优化的数据结构（SDS、跳跃表、压缩列表等）
  4. I/O多路复用：使用epoll等机制处理大量连接
  5. 编码优化：针对不同数据大小使用不同编码方式

3. 你知道Redis为什么要用Reactor模型吗？你项目中还有哪些组件也用了类似的设计

Reactor 模型就是一个“事件分发器”：它像服务员一样，统一接收所有客户的请求，然后派给工人去做，自己继续接待下一位，从而实现高并发、高性能的网络服务。

Reactor模型优势：

  • 高并发处理：单线程处理多个连接
  • 事件驱动：基于事件回调，非阻塞
  • 资源高效利用：避免为每个连接创建线程

项目中类似设计：

  • Netty：基于Reactor模式的网络框架
  • Nginx：事件驱动的Web服务器
  • Node.js：单线程事件循环模型

4. 讲讲io多路复用吧

I/O多路复用概念：

  • 一个进程/线程可以监听多个文件描述符
  • 当某个文件描述符就绪时，通知程序进行相应操作
  • 避免了为每个连接创建线程的开销

三种实现方式：

  1. select：跨平台，但效率较低
  2. poll：改进select，但仍有性能问题
  3. epoll：Linux特有，效率最高

5. 你们项目中为什么选用 RocketMQ？和 Kafka、RabbitMQ 相比你觉得它的优势在哪

RocketMQ优势：

  1. 消息可靠性：支持事务消息、顺序消息
  2. 性能优秀：单机支持10万级QPS
  3. 功能丰富：支持消息轨迹、消息重试、死信队列
  4. 运维友好：提供丰富的监控和管理工具
  5. 阿里生态：与阿里云服务集成良好

对比分析：

特性 RocketMQ Kafka RabbitMQ
消息可靠性 高 高 高
顺序消息 支持 支持 不支持
事务消息 支持 不支持 支持
性能 高 极高 中等
运维复杂度 中等 高 低

6. 消费时，Consumer和Consumer Group 的区别是什么

概念区别：

  • Consumer：单个消费者实例
  • Consumer Group：消费者组，包含多个Consumer实例

关键特性：

  1. 负载均衡：同一Group内的Consumer分摊消息
  2. 消费位点：Group级别维护消费进度
  3. 容错机制：Group内Consumer故障不影响整体消费

7. 如果一个topic有5个queue，那我起10个consumer会怎么样？都能用上吗

答案：不能都用上

原因分析：

  • RocketMQ的负载均衡策略是：一个Queue只能被一个Consumer消费
  • 5个Queue最多只能被5个Consumer消费
  • 剩余的5个Consumer会处于空闲状态

8. 什么是事务消息？RocketMQ怎么保证事务消息

事务消息概念：

  • 保证本地事务和消息发送的原子性
  • 解决分布式事务问题

实现机制：

  1. 半事务消息：先发送消息，但不立即投递
  2. 本地事务执行：执行本地业务逻辑
  3. 事务状态回查：根据本地事务结果决定消息是否投递

9. 如果本地事务成功了，但是commit消息时因为网络问题失败了，RocketMQ怎么处理？是谁去查谁

处理机制：

  1. Broker主动回查：Broker会定期回查Producer的本地事务状态
  2. Producer被动响应：Producer需要实现回查接口
  3. 最终一致性：通过回查机制保证消息最终投递

10. 什么是死信队列？和重试队列的区别？

死信队列（DLQ）：

  • 消息经过多次重试后仍然失败，进入死信队列
  • 用于存储无法正常处理的消息
  • 需要人工干预处理

重试队列：

  • 消息消费失败时的临时存储队列
  • 系统会自动重试消费
  • 重试次数达到上限后进入死信队列

12. 你put一条数据进 ES，立马用 match 查不到，但term 可以查到，问题在哪

问题原因：

  • match查询：会对查询词进行分词，然后查询倒排索引
  • term查询：精确匹配，不进行分词
  • 近实时性：ES默认1秒刷新一次，新数据可能还未建立倒排索引

13. 假如我用Redis做了一个排行榜，基于 ZSet 维护，某天发现榜单排名混乱且跳动频繁，你觉得可能是哪几类问题？

可能的问题：

  1. 并发更新问题：多个客户端同时更新分数
  2. 网络延迟：Redis集群节点间同步延迟
  3. 分数计算错误：业务逻辑计算分数有误
  4. 内存不足：Redis内存不足导致数据丢失
  5. 连接池问题：连接池配置不当导致连接不稳定

14. 数据库某张大表做范围查询经常慢，但加了索引依旧效果不好，你会从哪些维度分析

分析维度：

  1. 索引设计问题：索引列选择不当、复合索引顺序错误
  2. 查询语句问题：SELECT *、函数运算、隐式转换
  3. 数据分布问题：数据倾斜、基数过低
  4. 表结构问题：字段类型不匹配、冗余字段
  5. 数据库配置问题：缓冲池大小、查询缓存

代码示例：

15. 线程池中提交任务耗时越来越长，你发现核心线程数和阻塞队列都充足，那为什么还会慢

可能原因：

  1. 任务执行时间过长：单个任务耗时增加
  2. 资源竞争：CPU、内存、I/O资源不足
  3. 死锁或阻塞：任务间相互等待
  4. 外部依赖慢：数据库、网络调用延迟
  5. 垃圾回收：频繁GC导致暂停

17. 手撕：设计一个map，满足以下几点: 字符串表示键，整数表示值返回具有前缀等于给定字符串的键的值的总和

解题思路：

  • 使用Trie树（前缀树）存储字符串键
  • 每个节点记录以该前缀结尾的所有值的总和
  • 支持快速前缀查询和值更新

代码实现：

// Trie树节点
class TrieNode {
    Map<Character, TrieNode> children;
    int value; // 以当前节点结尾的键的值
    int prefixSum; // 以当前节点为前缀的所有值的总和
    
    public TrieNode() {
        this.children = new HashMap<>();
        this.value = 0;
        this.prefixSum = 0;
    }
}

// 前缀Map实现
public class PrefixMap {
    private TrieNode root;
    
    public PrefixMap() {
        this.root = new TrieNode();
    }
    
    // 插入或更新键值对
    public void put(String key, int value) {
        if (key == null) return;
        
        TrieNode current = root;
        int oldValue = get(key); // 获取旧值
        
        // 遍历字符串，构建或更新Trie树
        for (char c : key.toCharArray()) {
            current.children.putIfAbsent(c, new TrieNode());
            current = current.children.get(c);
            
            // 更新前缀和：减去旧值，加上新值
            current.prefixSum = current.prefixSum - oldValue + value;
        }
        
        // 设置当前节点的值
        current.value = value;
    }
    
    // 获取指定键的值
    public int get(String key) {
        if (key == null) return 0;
        
        TrieNode current = root;
        for (char c : key.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return 0;
            }
            current = current.children.get(c);
        }
        return current.value;
    }
    
    // 获取指定前缀的所有值的总和
    public int getPrefixSum(String prefix) {
        if (prefix == null) return 0;
        
        TrieNode current = root;
        for (char c : prefix.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return 0;
            }
            current = current.children.get(c);
        }
        return current.prefixSum;
    }
    
    // 删除指定键
    public void remove(String key) {
        if (key == null) return;
        
        int value = get(key);
        if (value == 0) return; // 键不存在
        
        TrieNode current = root;
        for (char c : key.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return;
            }
            current = current.children.get(c);
            current.prefixSum -= value; // 更新前缀和
