OAuth 2.0 概览

OAuth 2.0 是一套“授权”框架，用于在不暴露用户密码的前提下，让第三方应用代表资源所有者访问受保护资源。它关注“凭什么访问”和“访问到什么程度”，而非“你是谁”。

  • 参与方：

    • Resource Owner（资源所有者）：通常为用户。
    • Client（客户端应用）：请求访问资源的应用（Web/SPA/Native/Server）。
    • Authorization Server（授权服务器）：发放令牌并进行授权。
    • Resource Server（资源服务器）：托管受保护资源，校验令牌。

  • 核心概念：

    • Authorization Grant（授权许可）：获取令牌的方式（如授权码）。
    • Authorization Code（授权码）：短期一次性码，用于换取令牌。
    • Access Token（访问令牌）：调用资源接口的凭据，短期有效。
    • Refresh Token（刷新令牌）：用于换取新的访问令牌，通常仅在受信任环境下使用。
    • Scope（权限范围）：最小化授权原则，精确声明允许访问的资源/动作。
    • Redirect URI（重定向地址）：授权回调地址，必须严格匹配预注册列表。
    • State：CSRF 防护随机串，客户端发起授权时生成并校验。
    • PKCE：授权码流在公有客户端（SPA/Native）中的增强安全机制。

常见授权方式（Grants）

  1. 授权码模式（Authorization Code）+ PKCE（强烈推荐：Web SPA/Native）
  • 流程要点：
    1. 客户端生成 code_verifier 与 code_challenge（S256），连同 state 发起授权请求。
    2. 用户在授权服务器登录并同意授权后，浏览器重定向回携带 code 与原始 state。
    3. 客户端用 code + code_verifier 向授权服务器换取 access_token（可选 refresh_token）以及（若使用 OIDC）id_token。
    4. 客户端持 access_token 调用资源服务接口。
  • 安全性：避免令牌暴露在前端 URL，PKCE 防止授权码在中途被窃取后被滥用。
  1. 客户端凭证模式（Client Credentials）
  • 用于后端-后端（机器到机器）调用，无用户参与，按客户端身份与授权范围发令牌。
  1. 设备码模式（Device Code）
  • 适用于输入受限设备（如 TV、IoT）。设备获取 device_code；用户在另一个设备完成登录与授权；设备轮询换取令牌。
  1. 已不推荐/废弃场景
  • 隐式模式（Implicit）：令牌经浏览器 URL 直接下发，存在泄露风险，不再推荐。
  • 资源所有者密码模式（Password）：暴露用户密码给客户端，已被弃用。

标准端点与交互

  • /authorize：发起授权，返回授权码或错误。
  • /token：用授权码/客户端凭证/设备码换取 access_token（以及可选 refresh_token、id_token）。
  • /revocation：撤销令牌（Access/Refresh）。
  • /introspection：查询令牌状态与范围（多用于不透明令牌）。

访问令牌与刷新令牌

  • Access Token：短期、可撤销。可为不透明字符串，也可为 JWT（自包含声明）。
  • Refresh Token：仅在受信任环境使用，通常不下发给纯前端 SPA；建议启用刷新令牌轮换（Rotation）。
  • 令牌格式：
    • JWT：携带签名（JWS），可离线验证，有过期时间 exp、签发方 iss、受众 aud、签发时间 iat 等。
    • Opaque（不透明）：需调用授权服务器 introspection 才能解析状态，服务端更易撤销与集中管理。

OAuth 2.0 实践与安全要点

  • 始终使用授权码 + PKCE（SPA/Native）；后端使用客户端凭证。
  • 严格校验 Redirect URI，禁止通配与开放重定向。
  • 使用 state 防 CSRF；SPA/OIDC 再配合 nonce 防重放。
  • 最小化 Scope，按资源边界拆分，避免过度授权。
  • 短期 Access Token + 刷新令牌轮换，泄露时可快速失效；配合撤销与吊销机制。
  • HTTPS 全站强制，禁止明文传输令牌；避免在 URL/Referer 中泄露。
  • 前后端分离推荐：前端仅持 access_token（内存/Session Storage）；敏感刷新逻辑放后端 BFF；或使用 SameSite 严格 Cookie 搭配后端会话。
  • JWT 校验：校验证书（JWKs）、签名、exp/nbf、iss/aud，并考虑 jti 与黑名单机制。
  • CORS/回调白名单：限制可信来源；对第三方脚本谨慎。

OIDC（OpenID Connect）

OIDC 建立在 OAuth 2.0 之上，目标是“身份认证”。它定义了如何向客户端证明“用户是谁”，并提供标准化的用户信息获取方式。

  • 关键产物：ID Token

    • 格式通常为 JWT（JWS）
    • 面向客户端，声明用户身份信息（sub 唯一标识、iss、aud、exp、iat、auth_time、nonce 等）。
    • 客户端必须验证签名与关键声明：
      • iss 与授权服务器一致；
      • aud 包含本客户端 client_id；
      • exp 未过期；
      • 若使用 nonce，需与请求时一致（防重放）。

  • UserInfo Endpoint：

    • 通过 access_token 获取用户可读信息（name、email、picture 等）。
    • 受 Scope 控制（如 openid email profile）。

  • Discovery & JWKs：

    • 发现文档：/.well-known/openid-configuration
    • 包含授权端点、令牌端点、UserInfo、JWKS 地址、支持的算法与响应类型等。
    • JWKS（公钥集）用于校验 ID Token/JWT 的签名。

  • 常见 OIDC 流程（授权码 + PKCE）：

    1. 客户端带 scope=openid profile email、state、nonce、code_challenge 跳转授权。
    2. 用户登录并同意授权，回调附带 code、state。
    3. 客户端用 code + code_verifier 换取 access_token、id_token（可选 refresh_token）。
    4. 客户端本地验证 id_token 签名与声明；若需要更详细的用户资料，用 access_token 调 UserInfo。

OAuth 2.0 与 OIDC 的区别与选型

  • 定位：

    • OAuth 2.0：授权框架，解决“能否访问、可访问到什么”。
    • OIDC：基于 OAuth 2.0 的认证协议，解决“你是谁”。

  • 令牌：

    • OAuth 2.0：核心是 access_token（授权凭据）。
    • OIDC：引入 id_token（认证凭据，标识用户身份）。

  • 端点与文档：

    • OAuth 2.0：/authorize、/token、/revocation、/introspection。
    • OIDC：在此基础上增加 /.well-known/openid-configuration、/userinfo、/jwks 等。

  • 客户端关注点：

    • 若仅需代表用户访问资源（无登录态展示），选 OAuth 2.0 即可。
    • 若需要登录、用户身份与资料、单点登录（SSO），选 OIDC（openid Scope 必填）。

  • 推荐选型：

    • Web/SPA/Native 登录：OIDC 授权码 + PKCE。
    • 后端服务对接：OAuth 2.0 客户端凭证模式。
    • 设备/TV 登录：OAuth 2.0 设备码模式（可结合 OIDC 获取身份）。

常见问题与排查

  • 回调报错：检查 Redirect URI 完全匹配；确认 state 原样返回且被校验。
  • ID Token 校验失败：核对 iss/aud、时间偏差、签名算法与 JWKS 缓存策略。
  • 跨域问题：确认授权页与回调页域名、CORS 配置、SameSite 策略。
  • 刷新失败：是否启用轮换、刷新令牌有效期与撤销机制、客户端机密是否泄露。

术语速览

  • state：请求/回调关联与 CSRF 防护。
  • nonce：OIDC 防重放，仅与 id_token 相关。
  • aud/iss/sub：受众/发行者/用户唯一标识。
  • prompt/login_hint/max_age：控制交互体验与会话。
