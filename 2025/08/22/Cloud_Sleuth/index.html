背景与目标

在微服务架构中，请求会跨越多个服务、线程与消息中间件。要快速定位问题并衡量性能，需要“分布式链路追踪”。本文介绍两套主流方案：

  • 方案 A（Spring Boot 2.x / Spring Cloud 2021.x）：Spring Cloud Sleuth + Zipkin
  • 方案 B（Spring Boot 3.x）：Micrometer Tracing + Brave/OTel + Zipkin

给出可复制的依赖、配置、示例代码，演示如何在日志中自动打印 traceId/spanId、如何跨服务（RestTemplate/Feign）与消息队列（Kafka/RabbitMQ）透传上下文，并提供排错指南。

核心概念

  • traceId: 一次端到端请求的全局 ID。
  • spanId: 某个片段（如一次 HTTP 调用、数据库操作）的 ID。
  • parentId: 当前 span 的父 spanId。
  • 采样率 sampling: 控制采集比例，降低开销。
  • Baggage: 自定义业务键值对，随 trace 在各服务之间透传（如 tenantId、userId）。

方案选型速览

  • Spring Boot 2.6~2.7 + Spring Cloud 2021.x：继续使用 Sleuth（Brave 实现）最直接。
  • Spring Boot 3.x：Sleuth 已停止维护，推荐使用 Micrometer Tracing；API 相似，迁移较平滑。

下文两套方案都给出，可按你项目版本选择其一即可。

常见问题排查

  • Zipkin 收不到数据：
    • base-url/endpoint 是否正确；
    • 服务器是否能访问 Zipkin 9411 端口；
    • 采样率是否过低；
    • 应用是否真的产生了 HTTP/MQ 调用。
  • 日志没有 traceId：
    • 检查 logging.pattern.level 是否包含 %X{...}；
    • 日志实现使用 logback；
    • 控制台与文件 appender 是否复用了相同 pattern。
  • 跨线程/异步丢失上下文：
    • 优先使用框架管理的异步（如 @Async、@Scheduled），Sleuth/Micrometer 会包装线程池；
    • 自建线程池时使用 LazyTraceExecutor（Sleuth）或 ContextPropagatingExecutor（Micrometer）包装。
  • 网关/负载均衡：
    • 确保网关也接入追踪；
    • 外部入口的 traceId 会在网关生成并向后游透传。

建议的生产配置

  • 采样率 1%~10%，根据 QPS 与存储成本权衡。
  • 使用异步上报（默认）且对 Zipkin 设置超时与重试策略。
  • 日志接入统一平台（ELK/OpenSearch）；通过 traceId 聚合日志更高效。
  • 尽量在入口（网关/边缘服务）统一注入关键 Baggage 字段。
