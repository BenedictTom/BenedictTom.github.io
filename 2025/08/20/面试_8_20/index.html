什么是反射，有什么安全隐患

  • 概念：运行时检查/操作类、方法、字段、构造器，支持动态创建对象、调用方法、读写字段。
  • 优缺点：灵活但有性能损耗（方法句柄/缓存可缓解）。
  • 安全风险：突破封装（私有成员可见）、setAccessible 绕过检查、配合反序列化形成 RCE、破坏模块边界/类加载隔离。
  • 规避：最小权限原则、白名单/黑名单校验、禁用不必要的 setAccessible、利用 JPMS 限制开放、在网关层做反序列化类型白名单。

反射有哪些应用场景

  • IoC/DI 与注解扫描、AOP 与动态代理、ORM 映射、序列化/反序列化框架、SPI 插件发现、测试访问私有、通用工具库。

什么是序列化与反序列化

  • 序列化：对象→字节/文本（网络传输、缓存、持久化）。常见：JDK、JSON、Protobuf、Kryo、Hessian。
  • 反序列化：字节/文本→对象（重建对象图）。
  • 风险：不安全反序列化导致 gadget 链执行；应做类型白名单、禁用默认信任。

序列化和反序列化是深拷贝还是浅拷贝

  • 一般等价于“深拷贝”（重新构建对象图）。
  • 例外：static/transient 字段不参与；外部资源、单例、线程池等非纯数据对象不被复制；自定义 readResolve 可能返回共享实例。

线程池最大等待时间是什么

  • 线程池本身无“任务最大等待时间”参数。
  • 常见超时：
    • Future.get(timeout)：等待结果的最长时间。
    • 队列 offer(task, timeout)：提交任务的等待时间（需使用有界/阻塞队列并显式超时）。
    • awaitTermination(timeout)：关闭后等待线程池结束的时间。
  • keepAliveTime：空闲线程存活时间，不是任务等待时间。

线程池怎么区分核心线程和非核心线程

  • <= corePoolSize 为核心线程；(corePoolSize, maximumPoolSize] 为非核心线程。
  • 非核心线程空闲超过 keepAliveTime 会回收；allowCoreThreadTimeOut(true) 时核心也会超时回收。
  • 提交流程：先占满核心→入队→队列满且 < max 新建非核心→仍超出触发拒绝策略。

synchronized 和 Lock 有什么区别

  • 语法与释放：synchronized 关键字、自动释放；Lock（如 ReentrantLock）需手动 unlock。
  • 能力：Lock 支持可中断获取、可超时、公平锁、多个条件队列；synchronized 无上述扩展。
  • 可见性与重入：两者均保证互斥与可见性，均可重入。
  • 性能/场景：Lock 更灵活，适合复杂并发与高竞争；synchronized 简洁、JIT 优化好。

synchronized 加在普通方法和静态方法区别？

  • 普通方法：锁对象是实例（this），不同实例互不影响。
  • 静态方法：锁对象是类（Class），同一类所有实例互斥。

Lock 怎么实现公平和非公平的

  • ReentrantLock(boolean fair)：true 公平（FIFO 取锁），false 非公平（新线程可插队）。
  • 实现基础：AQS 的 CLH 队列管理获取顺序与唤醒。

@Transactional 注解什么时候失效

  • 非 public 方法；同类内自调用（未经过代理）；final/静态方法（无法代理）；
  • 捕获异常未抛出或抛出受检异常（默认仅 RuntimeException 回滚）；
  • 未被 Spring 管理/错误的代理方式；异步线程执行；数据源/事务管理器不匹配。

如何在类内调用另一个事务方法仍生效

  • 通过代理而非 this：
    • AopContext.currentProxy()（需 expose-proxy）。
    • 从 ApplicationContext 获取自身代理 Bean 调用。
    • 将目标方法拆到另一个 Bean，由容器注入后调用。

动态代理有哪几种

  • JDK 动态代理（基于接口）；CGLIB（基于子类）；ByteBuddy/Javassist（字节码生成）。
  • Spring：有接口优先 JDK，否则用 CGLIB。

@Resource 和 @Autowired 区别（name vs type）

  • @Autowired：按类型装配，可配 @Qualifier 指定名称；required 默认 true。
  • @Resource：JSR-250，优先按名称（字段名/显式 name），找不到再按类型。

如何使用 @Autowired 按名称注入

  • 在依赖点使用 @Qualifier(“beanName”)。
  • 或在候选上使用 @Primary 调整默认选择（仍是按类型后选优）。

如何解决循环依赖

  • 单例 Setter/字段注入：Spring 通过三级缓存提前暴露代理可解。
  • 构造器循环：默认无法解决，需拆分依赖/改 Setter/引入 @Lazy。
  • 原型作用域：无法自动解决，考虑改为单例或显式装配。

懒加载是什么

  • 延迟到真正使用时才创建/加载资源，降低启动/内存/IO 成本。
  • Spring：@Lazy；JPA：LAZY 关联需注意会话边界（避免 LazyInitializationException）。

事务隔离级别，分别解决了什么问题

  • 读未提交（Read Uncommitted）：可能脏读。
  • 读已提交（Read Committed）：避免脏读，仍有不可重复读/幻读。
  • 可重复读（Repeatable Read）：避免脏读/不可重复读；幻读取决于读类型与锁。
  • 串行化（Serializable）：避免三类异常，吞吐最低。

RR 级别如何解决幻读

  • InnoDB：
    • 快照读（普通 SELECT）：MVCC 版本控制避免幻读。
    • 当前读（SELECT … FOR UPDATE/UPDATE）：Next-Key Lock（记录锁+间隙锁）封锁范围，阻止插入幻行。

MySQL 中有哪几种锁

  • 粒度：表锁、行锁（记录锁、间隙锁、Next-Key 锁）。
  • 意向锁：IX/IS（与行锁共存，提升加锁效率）。用于解决表锁行锁通信问题
  • 元数据锁（MDL）：保护表结构变更；AUTO-INC 锁：自增值分配。
  • 语义：共享/排他、悲观/乐观（逻辑层）。

这个问题问得非常精准，已经触及到 InnoDB 锁机制的核心了！我们来用大白话+图解思维，把 行锁、记录锁、间隙锁、Next-Key 锁之间的关系彻底讲清楚。

🔑 一句话总结：

“行锁”是一个大类，它包括了“记录锁”和“间隙锁”，而 InnoDB 实际使用的是它们的组合：Next-Key Lock。

🧩 一、先搞清概念：它们的关系就像“水果”和“苹果、香蕉”

大类（广义） 具体类型（狭义）
行锁（Row-level Lock） ✅ 记录锁（Record Lock）
✅ 间隙锁（Gap Lock）
✅ Next-Key Lock（记录 + 间隙）

👉 所以：

  • “行锁”是统称，表示“锁某一行或某个范围”
  • “记录锁”、“间隙锁”是具体实现方式
  • InnoDB 的“行锁”其实是 Next-Key Lock（记录锁 + 间隙锁）

📚 二、三种锁分别是什么？（配合例子）

我们还是用电影院座位的例子：

1️⃣ 记录锁（Record Lock）

🔒 锁的是 已经存在的某一条记录

  • InnoDB 会给 id=5 这条记录加 记录锁（X锁）
  • 别人不能改、不能删 id=5
  • 但可以在 (3,5) 或 (5,8) 之间插入新座位（比如 4 或 6）

✅ 类比：你锁住了“5号座位”这把椅子，别人不能坐，但可以在空地建新椅子。

2️⃣ 间隙锁（Gap Lock）

🔒 锁的是 两个记录之间的“空隙”，不锁记录本身

  • InnoDB 会加一个 间隙锁：(3, 8)
  • 意思是：“不允许插入 id 在 (3,8) 范围内的新记录”
  • 所以 INSERT INTO seats (id=4)、id=6 都会被阻塞

✅ 类比：你在 3 号和 8 号之间拉了一条警戒线，写着“不准在这之间加新椅子！”

⚠️ 注意：间隙锁 不锁已存在的记录（比如 id=5），但通常会和记录锁一起用。

3️⃣ Next-Key Lock（临键锁）

🔒 是 记录锁 + 间隙锁 的组合，锁一个左开右闭的区间

InnoDB 默认使用的就是这种锁！

InnoDB 实际上会加多个 Next-Key Lock，比如：

  • 锁 (3, 5]：意思是“不允许插入 4 号，也不允许改 5 号”
  • 锁 (5, 8)：意思是“不允许插入 6、7 号”

✅ 类比：你不仅锁住了“5号椅子”，还锁住了“3到5之间的空地”，形成一个完整的保护圈。

🧭 三、关系图解

🧩 四、为什么需要组合？单独用不行吗？

单独用 问题 解决方案
只用记录锁 别人可以插入新记录 → 幻读 ❌ 不行
只用间隙锁 别人不能插，但能改已有记录 ❌ 不完整
记录锁 + 间隙锁（Next-Key） 既能防插入，又能防修改 ✅ 完美防止幻读

👉 所以 InnoDB 默认用 Next-Key Lock 来实现“可重复读”隔离级别下的幻读防护。

🚫 五、什么时候只用记录锁？（不加间隙）

在以下情况，InnoDB 只会加记录锁，不会加间隙锁：

✅ 场景1：等值查询 + 唯一索引 + 记录存在

  • 只需要锁 id=5 这一条记录
  • 不需要锁间隙，因为不可能插入另一个 id=5（唯一性保证）

👉 这时就是 纯记录锁

❌ 如果记录不存在，还是会加间隙锁！

✅ 六、总结：一张表说清关系

名称 锁什么 是否阻塞插入 是否阻塞修改 使用场景
记录锁（Record Lock） 已存在的某一行 ❌ 不阻塞插入 ✅ 阻塞修改/删除 精确查询（唯一索引）
间隙锁（Gap Lock） 记录之间的“空隙” ✅ 阻塞插入 ❌ 不阻塞修改（但通常配合记录锁） 范围查询，防幻读
Next-Key Lock 记录 + 左侧间隙 ✅ 阻塞插入 ✅ 阻塞修改 默认模式，防幻读
行锁（统称） 上面三者的总称 视情况而定 视情况而定 所有 DML 操作

💡 最后一句话总结：

“行锁”是总称，InnoDB 的“行锁”机制其实是“Next-Key Lock”——它由“记录锁”和“间隙锁”组成，既能锁住已存在的数据，又能锁住可能插入新数据的“空隙”，从而彻底防止幻读。

HTTP 与 HTTPS，HTTPS 如何保证安全

  • HTTPS = HTTP + TLS。
  • 机密性：对称加密；完整性：MAC/AEAD；身份认证：证书链与 CA。
  • 密钥交换：RSA/ECDHE（现代以 ECDHE 提供前向保密 PFS）。

三次握手过程

  • 客户端：SYN(seq=x) → 服务器：SYN+ACK(seq=y, ack=x+1) → 客户端：ACK(ack=y+1)，确认双方发送/接收能力与初始序列号。

握手期间可以传参数吗

  • 可以。TCP 选项：MSS、窗口扩大、SACK、时间戳等；
  • TLS 握手：SNI、ALPN、支持的密码套件/版本等。一般不携带应用层数据（TFO 可携带少量）。

二面

接口中可以定义哪些方法

  • 抽象方法、默认方法（default）、静态方法（static）、私有方法（Java 9+）；以及常量（public static final 字段）。

说一下包装类型，包装类型的缓存是什么

  • 缓存：Byte/Short/Integer/Long 为 [-128,127]；Boolean 两个常量；Character [0,127]。
  • Integer 上限可通过 JVM -XX:AutoBoxCacheMax 调整。

存储小数需要高精度用什么类型

  • BigDecimal（优先用 String 构造，明确舍入模式与精度）。

深拷贝和浅拷贝

  • 浅拷贝：复制引用；深拷贝：复制整个对象图。
  • 实现：序列化重建、拷贝构造/工厂、手写复制、重写 clone 并深度处理可变字段。

从 Caffeine 获取对象是深拷贝还是浅拷贝

  • 默认返回同一引用（浅拷贝语义）。如需隔离，存放不可变对象或在 get 后复制。

线程的状态

  • NEW、RUNNABLE（就绪/运行）、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。

sleep 和 wait 区别

  • 所属：sleep 在 Thread；wait 在 Object。
  • 锁：sleep 不释放锁；wait 释放锁并进入等待队列。
  • 唤醒：sleep 到时/中断；wait 需 notify/notifyAll 或超时。
  • 约束：wait 必须在同步块/方法内调用。

线程通信方式

  • wait/notify(ALL)、Lock+Condition、BlockingQueue、CountDownLatch、CyclicBarrier、Semaphore、Exchanger、join、volatile/原子类。

线程池参数，流程

  • 参数：corePoolSize、maximumPoolSize、keepAliveTime、TimeUnit、workQueue、threadFactory、RejectedExecutionHandler。
  • 流程：<core 新线程→否则入队→队列满且 <max 新建非核心→仍超出触发拒绝（Abort/CallerRuns/Discard/DiscardOldest）。

MySQL 索引类型

  • B+树（InnoDB 聚簇/二级）、哈希（MEMORY）、全文（InnoDB/MyISAM）、R-Tree（空间/GIS）。
  • 概念：前缀索引、联合索引、覆盖索引、最左前缀原则。

MyBatis 中 # 和 $ 区别

  • #{}：预编译占位，自动转义，防 SQL 注入，推荐用于值。
  • ${}：字符串拼接，用于动态对象名（表/列/排序），必须白名单校验。

自定义拦截器如何实现

  • Spring MVC：实现 HandlerInterceptor，注册到 WebMvcConfigurer#addInterceptors，配置路径与顺序。
  • 也可用 AOP 切面（@Aspect）实现横切关注点（日志、鉴权、埋点）。

线程池中提交的任务能取消吗

  • 能。保存 Future，调用 cancel(boolean mayInterruptIfRunning)。
  • 未执行：从队列移除；执行中：发送中断信号（任务需正确响应中断）。

spring为什么设置为默认单例

  • 大多数业务 Bean 无状态，单例复用降低创建与 GC 成本、占用更少内存。
  • 便于统一管理生命周期与横切能力（AOP、事务、事件、缓存等）。
  • 依赖注入、对象缓存友好，启动与运行期性能更好；且仅单例可通过三级缓存解决循环依赖。
  • 线程安全通过无状态/不可变设计保障；有状态场景可用 prototype 或 Web 作用域。

springboot 默认的动态代理是哪种，为什么

非常棒的问题！我们来澄清一个常见的误解。

❌ 常见误解：

“Spring Boot 默认是：有接口用 JDK 代理，没接口用 CGLIB”

这个说法是 ❌ 错误的！

✅ 正确答案：

Spring Boot 默认情况下，proxyTargetClass = true，即：无论有没有接口，都优先使用 CGLIB 代理！

证据来源：Spring Boot 源码

在 Spring Boot 中，@EnableAutoConfiguration 会导入一系列自动配置，其中 AOP 的默认配置是：

🔍 来源：org.springframework.boot.autoconfigure.aop.AopAutoConfiguration

看到关键点了没有？
@EnableAspectJAutoProxy(proxyTargetClass = true) —— 强制使用 CGLIB 代理！

🎯 所以 Spring Boot 的真实行为是：

条件 使用的代理方式
有接口（如 UserService） ✅ CGLIB 代理（生成子类）
没接口（普通类） ✅ CGLIB 代理（生成子类）

👉 统一使用 CGLIB，不再优先使用 JDK 动态代理！

🤔 为什么 Spring Boot 要这么做？

1. 避免代理类型混乱

  • 传统 Spring：有的 Bean 是 JDK 代理（$Proxy0），有的是 CGLIB 代理（$$EnhancerByCGLIB）。
  • 类型不一致，容易导致 ClassCastException，尤其是在注入实现类时。

2. 统一行为，减少坑

  • 开发者不需要关心“有没有接口”。
  • 所有 AOP（@Transactional, @Cacheable 等）都能生效，即使方法不在接口中。

3. 面向类编程更常见

  • 很多项目并不严格遵循“必须定义接口”的规范。
  • 直接在 Service 类上加注解更直观。

4. CGLIB 性能已足够好

  • 现代 JVM 对 CGLIB 的支持很好，启动时间和调用性能差异不大。
  • 宁愿牺牲一点性能，换取一致性和开发体验。

    • 尽量面向接口编程（利于测试、解耦）
    • 避免 final 类和方法（否则 CGLIB 无法代理）

循环依赖问题如何解决，为什么必须三级，一级二级为什么不行

🎯 问题背景：什么是循环依赖？

A 依赖 B，B 又依赖 A —— 形成循环依赖。

Spring 要在创建 A 的过程中注入 B，但在创建 B 的过程中又要注入 A……怎么办？

✅ Spring 的解决方案：三级缓存（3-level cache）

Spring 使用 三级缓存机制 来解决单例 Bean 的循环依赖问题。

三个缓存定义（DefaultSingletonBeanRegistry 中）：

缓存 类型 作用
一级缓存 singletonObjects 存放完全初始化好的 Bean（已创建、已填充属性、已初始化）
二级缓存 earlySingletonObjects 存放早期暴露的原始对象（未填充属性，未初始化）
三级缓存 singletonFactories 存放对象工厂（ObjectFactory），用于创建早期引用

🧱 为什么需要三级？不能只用一级或二级吗？

我们一步步推导：

❌ 方案 1：只用一级缓存（singletonObjects）—— 不行！

  • 一级缓存只存完全创建好的 Bean。
  • 但在循环依赖中，A 还没创建完（正在创建中），就要被 B 使用。
  • 如果只等 A 完全创建好再放一级缓存，那 B 就一直拿不到 A，死锁！

🔴 结论：一级缓存只能存“成品”，无法解决“半成品”被提前引用的问题。

❌ 方案 2：用一级 + 二级缓存（提前暴露原始对象）—— 在大多数情况下可以，但无法支持 AOP 代理！

假设我们这样做：

看起来没问题？✅
但问题来了：如果 A 需要被 AOP 代理（比如加了 @Transactional）怎么办？

🚨 核心问题：AOP 代理对象 vs 原始对象

假设 A 加了 @Transactional，Spring 应该返回一个 代理对象，而不是原始对象 new A()。

但在上面的流程中：

  • 我们提前把 原始对象 new A() 放入了二级缓存。
  • B 拿到的是 A 的原始对象，不是代理对象！
  • 即使后面 A 创建完了，生成了代理对象，B 里持有的还是旧的原始对象。

❌ 结果：B 依赖的 A 没有事务功能，AOP 失效！

✅ 正确方案：引入三级缓存（ObjectFactory）

Spring 的聪明之处在于：提前暴露的不是对象，而是一个“工厂”。

🌟 三级缓存的关键优势

优势 说明
✅ 支持 AOP 代理 通过 ObjectFactory，可以在暴露早期引用时动态决定是否返回代理对象
✅ 延迟代理创建 不在实例化后立即创建代理，而是等到被其他 Bean 依赖时才创建（懒加载）
✅ 保证引用一致性 后续所有对该 Bean 的早期引用，都通过同一个工厂生成，避免多次创建代理

🔁 为什么需要二级缓存？不能只用三级吗？

可以，但效率低。

  • 三级缓存中的 ObjectFactory 每次调用 getObject() 都可能重新创建对象（比如代理）。
  • 为了避免重复创建，Spring 将 ObjectFactory 生成的早期引用缓存到二级缓存。
  • 下次再获取时，直接从二级缓存取，不用再走工厂。

💡 二级缓存的作用：缓存 ObjectFactory 的执行结果，避免重复创建早期对象。

🧩 三级缓存的“替换”过程

  • 一旦 Bean 完全创建好，就从一、二、三级缓存中清理中间状态。
  • 最终只有一级缓存中有完整的 Bean。

✅ 总结：为什么必须是三级缓存？

方案 能否解决循环依赖 能否支持 AOP 代理 结论
只用一级缓存 ❌ ❌ 不行
一级 + 二级（放原始对象） ✅ ❌（拿不到代理） 不行
一级 + 二级 + 三级（ObjectFactory） ✅ ✅（动态生成代理） ✅ 必须这样

💡 核心原因：
为了在解决循环依赖的同时，保证 AOP 代理的正确性。
只有通过 ObjectFactory（三级缓存），才能在暴露早期引用时，动态决定返回“原始对象”还是“代理对象”。

📚 一句话总结

Spring 用三级缓存，不是为了“解决循环依赖”，而是为了“在解决循环依赖的同时，还能正确生成 AOP 代理对象”。
如果没有 AOP，二级缓存就够了；但因为有 AOP，必须用三级缓存来“延迟决定”返回什么引用。
