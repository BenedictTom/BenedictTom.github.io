CAS

CAS（Compare-and-Swap）是一种重要的原子操作，广泛应用于多线程编程中以实现同步机制。它在低级别并发控制中起着关键作用，主要用于确保在多线程环境下的数据一致性。

工作原理

CAS操作涉及三个参数：存储位置（V）、预期旧值（A）和新值（B）。其工作流程如下：

  1. 比较：检查存储位置V处的值是否与预期旧值A相等。
  2. 交换：如果相等，则将存储位置V的值更新为新值B，并返回成功；如果不相等，则不进行任何操作，并返回失败。

这一过程是原子性的，意味着它是不可分割的操作，在执行过程中不会被其他线程干扰。

应用场景

  • 无锁编程：CAS常用于实现高效的无锁数据结构，如无锁栈、队列等。通过使用CAS，可以避免传统的锁机制带来的性能开销和死锁问题。
  • 乐观锁：在数据库或分布式系统中，CAS思想可用于实现乐观锁机制，允许事务在提交时检查数据是否已被其他事务修改。

优点与挑战

  • 优点：相比于传统锁机制，CAS提供了更细粒度的并发控制，能够显著提高程序的并发性能。
  • 挑战：CAS可能导致ABA问题，即一个值从A变到B再变回A，使得CAS操作误认为没有变化发生。解决方法之一是引入版本号机制。

在Java中，原子类位于java.util.concurrent.atomic包下，提供了无锁、线程安全的操作，主要用于实现高效的并发控制。这些类通过底层硬件提供的CAS（Compare and Swap）操作来确保原子性，从而避免了传统锁机制带来的性能开销和复杂性。

常见的原子类

  1. AtomicInteger, AtomicLong, AtomicBoolean

    • 提供对基本整型、长整型以及布尔值的原子操作。
    • API示例：get(), set(), lazySet(), compareAndSet(expected, update), getAndIncrement(), incrementAndGet()等。

  2. AtomicReference

    • 对对象引用进行原子更新。
    • API与上述类似，但操作的是对象引用而非基本类型。

  3. AtomicIntegerArray, AtomicLongArray, AtomicReferenceArray

    • 支持对数组元素的原子操作。
    • API包括get(index), set(index, newValue), compareAndSet(index, expected, update)等。

  4. AtomicIntegerFieldUpdater, AtomicLongFieldUpdater, AtomicReferenceFieldUpdater

    • 允许以原子方式更新指定对象的指定字段。
    • 使用时需注意内存可见性和正确同步。

  5. StampedLock

    • 尽管不是原子类，但它提供了一种新的读写锁策略，支持乐观读取模式，适用于某些场景下的高并发需求。

原理说明

这些原子类主要依靠Java的Unsafe类来执行低级别的、直接的硬件级操作，如CAS。Unsafe提供了一系列方法用于直接操作内存中的数据，绕过了Java的普通访问控制，从而能够实现非常高效的原子操作。

  • CAS操作：核心是compareAndSwap方法，它尝试更新一个变量，但如果当前值与预期值不匹配，则不会进行更新，并返回失败。这种方法允许在没有锁定的情况下进行非阻塞的同步。

  • ABA问题：由于CAS仅检查值是否改变，而不关心值的变化过程（比如从A变到B再变回A），这可能导致潜在的问题。部分原子类（如AtomicStampedReference）通过引入版本号或时间戳来解决这个问题。

使用原子类可以显著提高多线程程序的性能，尤其是在高度竞争的环境中。不过，正确地应用它们需要理解其工作原理及局限性。例如，在复杂的依赖关系或需要复合操作时，可能仍需借助于更高层次的同步机制。

Java 中的原子类（java.util.concurrent.atomic 包下的类）适用于需要高效、无锁、线程安全地操作共享变量的场景。它们通过 CAS（Compare and Swap）机制 实现了对变量的原子更新，避免了传统 synchronized 锁带来的性能开销和上下文切换问题。

LongAddr

LongAdder 是 Java 8 引入的一个类，位于 java.util.concurrent.atomic 包中。它专门用于在多线程环境下高效地进行计数操作，是 AtomicLong 的一种替代方案，特别适用于高并发的场景。

主要特点

  1. 高并发性能：与 AtomicLong 相比，LongAdder 在高竞争（即大量线程同时尝试更新同一个计数器）的情况下表现得更加高效。这是因为 LongAdder 采用了分段锁的技术，内部维护了一个或多个变量（根据竞争程度动态调整），从而减少了线程冲突的可能性。

  2. 最终一致性：LongAdder 提供的是最终一致性模型，这意味着在并发更新期间，读取到的数据可能不是最新的值，但最终会收敛到正确的结果。这对于某些可以接受一定延迟的应用场景来说是非常合适的。

  3. 适合累加统计：LongAdder 特别适合于需要频繁进行递增操作的场合，比如统计访问次数、任务完成数量等。

常用方法

  • void add(long x)：将指定值加到当前值上。
  • long sum()：返回所有内部计数器的总和。注意，在高并发情况下，这个总和可能不精确，因为它代表调用时刻的状态。
  • void reset()：重置计数器的值为零。
  • long sumThenReset()：获取当前总和然后重置计数器，这是一个原子操作。

使用示例

适用场景 vs AtomicLong

  • 高并发环境下的累加统计：当有大量的线程并发执行对同一个计数器的递增操作时，LongAdder 更具优势。
  • 性能考虑：如果您的应用场景允许最终一致性，并且主要关注的是写入性能而非即时读取准确性，那么 LongAdder 是更好的选择。
  • 低竞争场景：对于竞争较低的场景，AtomicLong 可能更简单直接，因为它的开销较小。

总之，LongAdder 提供了一种高效的解决方案来处理高并发环境下的计数问题，尤其适用于那些不需要实时准确性的累加统计场景。

UNsafe

Unsafe 是 Java 提供的一个底层工具类，位于 sun.misc 包中（在较新版本的 JDK 中，它被迁移至 jdk.internal.misc）。尽管它不是公开的 API，不建议直接使用，但由于其提供的功能对于某些高性能或特定场景下的需求非常重要，因此在一些高级应用、并发库以及JVM内部实现中经常能看到它的身影。以下是对 Unsafe 的详细介绍。

主要特性

  1. 内存操作：

    • Unsafe 允许程序直接访问和修改内存中的数据，这包括分配新的内存区域、释放内存、读取和写入指定地址的数据等。
    • 例如：allocateMemory(long bytes), freeMemory(long address), getInt(Object obj, long offset) 等方法提供了对原始内存的直接控制。

  2. CAS 操作：

    • Unsafe 提供了 compareAndSwap (CAS) 方法，这是无锁算法的核心，用于实现高效的并发控制。通过 CAS 操作可以避免传统的锁机制带来的性能损耗。
    • 如：compareAndSwapInt(Object o, long offset, int expected, int x) 可以原子地更新对象字段值。

  3. 类实例化与字段访问：

    • 能够绕过构造函数直接创建对象实例，这对于某些需要特殊初始化逻辑的情况非常有用。
    • 同时，它还允许访问对象的私有字段，甚至是静态字段，而不需要反射。
    • 示例方法：allocateInstance(Class<?> cls), objectFieldOffset(Field f)。

  4. 线程调度：

    • 提供了一些与线程调度相关的功能，比如暂停和恢复线程执行，这对调试工具或者需要精细控制线程行为的应用很有帮助。
    • 相关方法如：park(boolean isAbsolute, long time) 和 unpark(Thread thread) 分别用于阻塞和唤醒线程。

  5. 数组操作：

    • 支持获取数组元素的基础偏移量和规模，这对于优化数组处理逻辑至关重要。
    • 方法示例：arrayBaseOffset(Class<?> arrayClass), arrayIndexScale(Class<?> arrayClass)。

使用注意事项

  • 安全性问题：由于 Unsafe 提供了对 JVM 内部结构的直接访问权限，错误使用可能导致不可预测的行为甚至系统崩溃。因此，除非绝对必要，并且开发者对 JVM 内部原理有足够的了解，否则应避免使用。

  • 兼容性风险：Unsafe 并不是官方标准的一部分，这意味着不同版本的 JDK 或者不同的 JVM 实现之间可能存在差异，从而影响代码的可移植性。

  • 封装与替代方案：为了减少直接依赖 Unsafe 带来的风险，许多现代 Java 库和框架（如 Netty、Hibernate 等）会将对 Unsafe 的调用封装起来，仅在其内部使用。此外，随着 Java 的发展，越来越多的原生支持（比如 VarHandles 在 Java 9 中引入）正在逐渐取代部分 Unsafe 功能。

final关键字在Java中的使用涉及编译时和运行时的不同层面的行为，其底层原理主要体现在以下几个方面：

编译期检查

  1. 变量声明：当一个变量被声明为final时，编译器会确保该变量只能被赋值一次。如果尝试多次赋值，则会在编译阶段报错。对于基本类型的final变量，这意味着一旦赋值后就不能再改变其值。

  2. 方法覆盖：如果一个方法被声明为final，那么子类将无法重写这个方法。这也是在编译期间进行的检查，确保了方法的行为不会被子类修改。

  3. 类定义：如果一个类被声明为final，则该类不能有任何子类。这同样是编译时的限制，防止了继承的发生。

运行时行为

  • 对象引用不可变性：对于引用类型的final变量，虽然引用本身不能指向其他对象（即不能重新赋值），但是通过这个引用来访问的对象的内容是可以改变的。因此，为了保证真正的不可变性，通常还需要确保引用的对象本身也是不可变的。

  • JVM层面的支持：

    • 在字节码级别，final字段会被标记为特殊的标志位，这样JVM就知道这些字段是最终的，不能被再次赋值。
    • 对于final方法，JVM在加载类时会识别出哪些方法是final的，并且优化方法调用路径，因为知道这些方法不会被重写，可以进行更高效的调用优化。
    • 当一个类被声明为final时，JVM知道它不会有子类，因此可以做出一些假设来优化性能，例如避免动态绑定带来的开销。

特殊情况

  • 匿名内部类和局部类中的final变量：在早期版本的Java中（直到Java 8之前），如果你希望在一个匿名内部类或局部类中使用外部方法的局部变量，那么这个变量必须被声明为final。这是因为这些类实际上是在运行时创建的独立实例，它们需要捕获并持有外部变量的一个副本。从Java 8开始，这一要求放宽为“effectively final”，即如果一个局部变量事实上没有被修改过，即使没有显式地声明为final，也可以被内部类使用。

内存模型与线程安全

  • final字段初始化的安全性：根据Java内存模型的规定，正确构造的对象可以通过发布给其它线程而无需同步，只要所有final字段都被正确初始化。这意味着，在对象构造完成后，任何线程看到的对象状态都将反映final字段的最终值，而不是部分初始化的状态。这是通过JIT编译器和JVM对final字段的特殊处理实现的，确保了即使在多线程环境下也能保持数据的一致性和可见性。

总结来说，final关键字不仅影响代码的编写方式，还在编译期和运行时提供了特定的行为和保障。它帮助开发者强制执行某些设计决策，如不可变性、防止方法重写等，并且在JVM层面得到了支持以优化性能和增强安全性。
