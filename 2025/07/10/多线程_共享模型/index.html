多线程_共享模型

Joie2025-07-102025-07-14

JMM，即Java Memory Model（Java内存模型），是Java语言规范中定义的一种用于多线程环境下如何正确地进行内存访问的机制。它旨在解决由于计算机系统中多级存储、多处理器和编译器优化等带来的可见性和重排序问题，确保程序在各种硬件和编译器优化下的执行结果与按照顺序一致性模型执行的结果一致。

主要目标

  1. 可见性：当一个线程修改了共享变量的值，其他线程能够立即看到这个修改。JMM通过happens-before原则保证某些操作之间的可见性。

  2. 原子性：一些基本类型变量的操作被认为是原子性的（例如对32位数据类型的读写操作）。此外，Java提供了更高层次的原子类（如AtomicInteger）来支持更复杂的原子操作。

  3. 有序性：防止编译器和处理器对指令进行重排序从而影响程序的正确性。虽然重排序可以提高程序性能，但也可能导致意外的数据竞争和错误。JMM通过禁止特定类型的重排序来维护程序的正确性。

Happens-Before原则

为了保证上述的目标，Java定义了happens-before原则来规定哪些操作先行发生于其他操作，并且这些操作的结果对于后续的操作是可见的。主要规则包括：

  • 程序顺序规则：一个线程中的每个操作，happens-before于该线程中的任意后续操作。
  • 监视器锁规则：对一个监视器的解锁，happens-before于随后对同一个监视器的加锁。
  • Volatile变量规则：对一个volatile域的写，happens-before于任意后续对同一个volatile域的读。
  • 线程启动规则：Thread对象的start()方法的调用，happens-before于被启动的线程中的任何动作。
  • 线程结束规则：线程中的所有操作，happens-before于其他线程检测到该线程已经结束（通过Thread.join()等方法）。
  • 中断规则：一个线程调用另一个线程的interrupt()方法时，happens-before于被中断线程检测到中断事件的发生。
  • 终结器规则：一个对象的构造函数执行结束happens-before于其finalizer的开始。
  • 传递性：如果A happens-before B, 且B happens-before C, 那么A happens-before C。

好的！我们来通俗地讲一讲 Happens-Before 原则，不讲技术术语，用生活中的例子来类比，帮助你理解。

🌰 举个生活中的例子：

想象你和朋友在写一本合作的小说。你们每人负责一部分内容，但有时候需要交换信息或确认某些情节。

现在问题是：
你写了一段话（比如“主角走进了房间”），然后发给朋友。朋友看到后接着写：“他发现桌子上有封信”。
但是，如果朋友没看到你写的那段话，他就直接写了后面的内容，可能会出错——比如他写的是“他在外面跑步”，那就对不上了。

所以你要确保：你先写了那段话，朋友看到了，然后再继续写后面的剧情。这就涉及到两个操作之间的顺序和可见性。

📚 那 Happens-Before 是什么？

Happens-Before 就是 Java 给你的一种保证：

A 操作 happens-before B 操作，意味着 A 的结果对 B 来说是可见的，并且 B 一定是在 A 之后执行的。

你可以把它理解成一个“先后顺序 + 结果可见”的承诺。

✅ 程序顺序规则：

在一个线程里，代码是按顺序执行的。

✅ 锁规则（synchronized）：

如果你加锁了，那么解锁之前做的所有操作，其他线程拿到锁后都能看到。

✅ Volatile 变量规则：

一个 volatile 变量的写操作，happens-before 后续对该变量的读操作。

比如：

当线程2读到 flag == true，它能看到线程1在此之前做的所有事情。

👉 类比：你把某个关键剧情标记为“必须立刻同步”，别人一看到这个剧情就知道整个故事更新了。

✅ 线程启动规则：

调用 thread.start() 之前的所有操作，都会 happen-before 线程内部的操作。

比如你在主线程设置了一些参数，再启动子线程，子线程能看见这些参数。

✅ 线程结束规则：

线程里的所有操作，都 happen-before 其他线程知道它结束了（比如通过 join()）。

👉 类比：等演员演完了整场戏，观众才知道剧情发展完了，可以开始讨论结局。

🧠 总结一下（一句话）：

Happens-Before 就是 Java 用来告诉你：“哪些操作的结果，别的线程一定能看得见；哪些动作一定是先于别的动作发生的。”

它的作用就是防止多线程环境下出现数据混乱、看不到最新值等问题。

如果你想记住它，可以这样想：

“A happens-before B”，意思是：B 能看到 A 做的一切改动，而且不会乱序执行。

可见性

在Java多线程编程中，可见性是一个非常重要的概念。它指的是当一个线程修改了共享变量的值之后，其他线程能否看到这个修改后的值。默认情况下，Java内存模型（JMM, Java Memory Model）允许编译器和处理器对指令进行重排序优化，同时每个线程还可以拥有自己的工作内存（本地缓存），这可能导致线程间的可见性问题。

可见性问题的原因

  1. 线程的工作内存：每个线程都有自己的工作内存，在该内存中保存着主内存中变量的副本。如果一个线程更新了某个变量的值，但没有将这个更新写回到主内存中，那么其他线程可能看不到这个更新。
  2. 指令重排序：为了提高执行效率，编译器和处理器可能会调整指令的执行顺序。这种重排序不会影响单线程程序的执行结果，但在多线程环境下可能导致其他线程看到未预期的结果。

解决可见性的方法

Java提供了几种机制来确保变量修改的可见性：

1. 使用 volatile 关键字

  • 当一个变量被声明为 volatile 后，任何对该变量的读写操作都会直接作用于主内存，跳过线程的工作内存。这意味着使用 volatile 变量可以保证变量的可见性。
  • 注意，volatile 并不能保证复合操作的原子性，比如 i++ 这样的操作需要额外的同步措施。

2. 使用 synchronized 关键字

  • 每次进入 synchronized 块之前，线程必须获取对象的锁，并且会清空工作内存中的变量值，强制从主内存加载最新值；退出 synchronized 块时，线程会将工作内存中的变量值刷新回主内存。
  • 这样做不仅保证了操作的原子性，同时也确保了可见性。

3. 使用显式锁 (Lock)

  • 类似于 synchronized，但是提供了更灵活的锁定机制，如可中断锁、定时锁等。
  • 使用 ReentrantLock 或其他实现 Lock 接口的类同样可以确保可见性。

4. 使用 final 关键字

  • 对于不可变的对象（即用 final 修饰的对象），它们一旦构造完成就不会再发生变化，因此不需要担心可见性问题。
  • 这是因为 JVM 在构造对象期间遵循特定规则，确保对象在构造完成后对其它线程是可见的。

原子性

在Java并发编程中，**原子性（Atomicity）**是指一个操作要么完全执行，要么完全不执行，不存在部分执行的状态。换句话说，在多线程环境下，一个原子操作不会被其他线程中断或干扰，从而保证了数据的一致性和完整性。

原子性的挑战

在多线程环境中，确保操作的原子性是具有挑战性的，因为：

  • 线程切换：操作系统可以在任何时间点暂停一个线程并切换到另一个线程，如果在这个过程中涉及到共享资源的操作没有保护措施，则可能导致数据不一致。
  • 指令重排序：编译器和处理器为了优化性能可能会对指令进行重排序，这可能会影响程序的预期行为，尤其是在涉及多个步骤完成的操作时。
  • 缓存一致性问题：每个CPU核心都有自己的缓存，当多个线程同时读写同一块内存区域时，如果没有适当的同步机制，可能会导致某些线程看到过期的数据。

实现原子性的方法

Java提供了多种方式来实现原子性，主要包括以下几种：

1. 使用 synchronized 关键字

  • synchronized 可以用来修饰方法或代码块，它能够保证在同一时刻只有一个线程可以执行被其保护的代码段，从而实现了原子性。
  • 当一个线程进入同步块之前，它必须获得对象的锁；退出同步块时会自动释放锁，并将工作内存中的变量值刷新回主内存，确保其他线程可以看到最新的状态。

2. 使用显式锁 (Lock)

  • 类似于 synchronized，但是提供了更灵活的锁定机制，如可中断锁、定时锁等。
  • ReentrantLock 是最常用的显式锁实现之一，它可以提供与 synchronized 相同的功能，但更加灵活。

3. 使用 java.util.concurrent.atomic 包中的类

  • Java提供了AtomicInteger, AtomicLong, AtomicBoolean, AtomicReference等类，它们支持无锁的线程安全编程模型。
  • 这些类通过使用硬件级别的原子指令（如CAS，Compare-And-Swap）来实现原子性，通常比使用锁的方式更高效。

4. 使用 java.util.concurrent 中的高级工具

  • Java还提供了一些高级并发工具，如CountDownLatch, CyclicBarrier, Semaphore等，它们可以帮助构建复杂的并发控制逻辑，同时也保证了操作的原子性。

总结

原子性是并发编程中的一个重要概念，确保了多线程环境下的数据一致性和可靠性。Java提供了多种机制来实现原子性，包括但不限于 synchronized、显式锁(Lock)、java.util.concurrent.atomic 包中的原子类以及各种高级并发工具。选择合适的方法取决于具体的应用场景及性能需求。理解这些机制的工作原理有助于编写高效且正确的并发程序。

在Java并发编程中，**原子性（Atomicity）和可见性（Visibility）**是两个非常重要的概念，它们各自解决不同的问题，并且经常需要一起考虑以确保多线程程序的正确性和效率。

原子性（Atomicity）

定义

  • 原子性指的是一个操作要么完全执行，要么完全不执行。换句话说，在执行一系列步骤时，如果这些步骤作为一个整体被视为原子操作，则不能被其他线程看到部分完成的状态。
  • 例如，i++ 这样的操作实际上包含三个步骤：读取 i 的值、将 i 的值加1、将新的值写回 i。如果在多线程环境下没有适当的同步措施，这个操作就不是原子性的，可能会导致数据不一致的问题。

实现方式

  • synchronized: 使用同步方法或同步代码块可以保证一段代码的原子性。
  • 显式锁 (Lock): 类似于 synchronized，但是提供了更灵活的锁定机制。
  • 原子类 (java.util.concurrent.atomic): 如 AtomicInteger, AtomicLong, AtomicBoolean, AtomicReference 等，通过硬件级别的原子指令（如CAS，Compare-And-Swap）来实现特定操作的原子性。

可见性（Visibility）

定义

  • 可见性关注的是当一个线程修改了共享变量的值之后，其他线程能否看到这个修改后的值。默认情况下，每个线程都有自己的工作内存（本地缓存），其中存储了主内存中的变量副本。如果一个线程更新了某个变量的值，但没有将这个更新写回到主内存中，那么其他线程可能看不到这个更新。
  • 可见性问题通常发生在多线程环境下，特别是在使用非原子操作或缺乏适当同步的情况下。

实现方式

  • volatile 关键字: 当一个变量被声明为 volatile 后，任何对该变量的读写操作都会直接作用于主内存，跳过线程的工作内存，从而保证了可见性。
  • synchronized: 每次进入同步块之前，线程必须获取对象的锁，并且会清空工作内存中的变量值，强制从主内存加载最新值；退出同步块时，线程会将工作内存中的变量值刷新回主内存。
  • final 关键字: 对于不可变的对象（即用 final 修饰的对象），它们一旦构造完成就不会再发生变化，因此不需要担心可见性问题。

区别与联系

特性 原子性 可见性
定义 保证一个操作要么完全执行，要么完全不执行，中间状态对外不可见。 确保一个线程对共享变量所做的修改能够被其他线程看见。
关注点 多个步骤作为一个整体执行，防止部分执行带来的数据一致性问题。 数据的一致性和及时性，确保所有线程都能看到最新的数据状态。
解决方案 synchronized, 显式锁, 原子类等。 volatile, synchronized, 显式锁, final 等。

结合使用

在实际开发中，原子性和可见性通常是需要同时考虑的：

  • synchronized 和 显式锁 (Lock) 不仅能保证原子性，还能保证可见性。因为当一个线程释放锁时，它会将工作内存中的变化刷新回主内存；而当另一个线程获取同一个锁时，它会从主内存中重新加载最新的值。
  • volatile 主要用于保证可见性，但它并不能保证复合操作的原子性。例如，虽然 volatile int count 能够保证每次对 count 的读写都是立即反映到主内存的，但它并不能保证类似 count++ 这样的复合操作是原子性的。
