面试记录 2025-9-23

📌 候选人信息

  • 姓名：xxx
  • 技术方向：Java后端开发（具备Python能力）
  • 工作年限：11年（2014.08-2025.02）

✅ 资深 Java 工程师（11年经验）面试题库

总时长：35分钟

结构：基础 → 项目 → 编码

一、Java 核心与高并发（10分钟）

1.1 ThreadPoolExecutor 的核心参数？如果任务过多，队列满了会怎样？

参考答案：

  • 核心参数：

    • corePoolSize：核心线程数，常驻内存
    • maximumPoolSize：最大线程数（临时线程）
    • keepAliveTime + unit：非核心线程空闲存活时间
    • workQueue：任务队列（如 LinkedBlockingQueue）
    • threadFactory：线程创建工厂
    • handler：拒绝策略

  • 队列满后的处理流程：

    1. 当前线程数 < maximumPoolSize → 创建新线程执行任务
    2. 已达 maximumPoolSize 且队列满 → 触发拒绝策略：
      • AbortPolicy（默认）：抛出 RejectedExecutionException
      • CallerRunsPolicy：由提交任务的线程直接执行（降级）
      • DiscardPolicy：静默丢弃
      • DiscardOldestPolicy：丢弃队列最老任务，重试提交

✅ 加分点：提到“线上推荐使用 CallerRunsPolicy 防止雪崩”

1.2 简述 JVM 内存结构？各区域作用？

参考答案：

JVM 内存分为：

区域 作用 是否线程私有
程序计数器 记录当前线程执行字节码的位置 ✅ 是
虚拟机栈 存放方法调用栈帧（局部变量、操作数栈） ✅ 是
本地方法栈 Native 方法调用 ✅ 是
堆（Heap） 对象实例、数组存储区域 ❌ 否
方法区（元空间） 类信息、常量、静态变量、JIT 编译代码 ❌ 否

✅ 注意：JDK 8 后“永久代”被“元空间（Metaspace）”取代，使用本地内存。

1.3 ThreadLocal 内存泄露的原因？如何避免？

参考答案：

  • 原因：

    • ThreadLocal 的 Entry 是 WeakReference<K>，但 value 是强引用
    • 线程长期运行（如线程池），ThreadLocal 被回收后，value 仍被 Thread 的 threadLocals map 引用 → 无法回收 → 内存泄露

  • 解决方案：

    1. 每次使用后手动调用 remove()：
    2. 使用 try-finally 或 AutoCloseable 封装
    3. 避免 static ThreadLocal 长期持有

✅ 加分点：提到“Tomcat 中因未 remove 导致 PermGen 溢出”的真实案例

1.4 ConcurrentHashMap 如何实现线程安全？JDK 7 和 8 的区别？

参考答案：

  • JDK 7：

    • 分段锁（Segment），每个 Segment 相当于一个 ReentrantLock + HashEntry[]
    • 并发度 = Segment 数（默认 16）
    • 锁粒度大，性能一般

  • JDK 8：

    • 放弃分段锁，使用 synchronized + CAS + volatile
    • Node 数组 + 链表/红黑树
    • 锁粒度降到单个桶（bin）
    • 插入时用 CAS 尝试，失败再 synchronized

✅ 优势：JDK 8 更高效，尤其在低冲突场景

1.5 synchronized 和 ReentrantLock 的区别？

对比项 synchronized ReentrantLock
实现 JVM 层面（Monitor） JDK 层面（AQS）
灵活性 低（自动加锁/释放） 高（可中断、超时、公平锁）
公平性 非公平 可设置公平锁
条件变量 wait/notify Condition（更灵活）
锁获取信息 无法判断是否持有锁 可通过 isHeldByCurrentThread() 判断

✅ 使用建议：

  • 简单场景 → synchronized
  • 复杂控制（如超时、中断）→ ReentrantLock

1.6 AOP 是如何实现的？JDK 动态代理和 CGLIB 有何区别？

参考答案：

  • AOP 实现原理：

    • Spring AOP 基于动态代理
    • 运行时生成代理对象，织入通知（Advice）

  • JDK 动态代理 vs CGLIB：

对比项 JDK 动态代理 CGLIB
原理 实现接口的代理 继承目标类生成子类
要求 目标类必须实现接口 无接口也可代理
性能 稍慢（反射） 稍快（ASM 字节码）
final 方法 无法代理 也无法代理

✅ Spring 默认优先用 JDK 代理，无接口时用 CGLIB

二、数据库与缓存（8分钟）

2.1 MySQL 事务的特性（ACID）？在 Spring 中事务失效的几种情况？

参考答案：

  • ACID：

    • Atomicity：原子性（全成功或全失败）
    • Consistency：一致性（状态合法）
    • Isolation：隔离性（并发控制）
    • Durability：持久性（写入磁盘）

  • Spring 事务失效场景：

    1. 方法非 public：@Transactional 只对 public 方法有效
    2. 自调用：同类中 this.method() 调用，绕过代理
    3. 异常被捕获未抛出：try-catch 吃掉异常
    4. 异常类型不匹配：默认只回滚 RuntimeException
    5. 数据源未配置事务管理器
    6. Propagation 配置错误：如 NOT_SUPPORTED

✅ 解决：使用 AopContext.currentProxy() 或重构调用逻辑

2.2 MySQL 索引的设计思路？什么是覆盖索引？如何优化慢查询？

参考答案：

  • 索引设计原则：

    • 高频查询字段建索引
    • 联合索引遵循最左前缀
    • 避免过多索引（影响写性能）
    • 选择性高的字段优先（如 status 不适合单独建索引）

  • 覆盖索引（Covering Index）：

    • 查询字段全部包含在索引中，无需回表
    • 例如：idx_name_age(name, age)，查询 SELECT name, age FROM t WHERE name=? 可覆盖

  • 慢查询优化步骤：

    1. 开启 slow_query_log
    2. 用 EXPLAIN 分析执行计划
    3. 检查是否走索引、有无全表扫描、type=ALL
    4. 优化 SQL 或添加索引
    5. 必要时分库分表

2.3 Redis 缓存穿透、击穿、雪崩？你们项目中如何应对？

参考答案：

问题 定义 解决方案
穿透 查 DB 不存在的数据，反复打到底层 ✅ 缓存空值 + ✅ 布隆过滤器
击穿 热点 key 过期瞬间，大量请求打穿到 DB ✅ 互斥锁（Redisson）重建缓存
雪崩 大量 key 同时过期，DB 瞬间压力激增 ✅ 随机过期时间 + ✅ 多级缓存 + ✅ 降级

✅ 项目实践（华贸云）：

  • 布隆过滤器拦截非法商品 ID 查询
  • Redisson 分布式锁防止“开票”热点 key 被击穿
  • 缓存过期时间加随机值（±300s）

2.4 如何实现分布式锁？Redis 和 ZooKeeper 方案对比？

参考答案：

  • Redis 实现：

    • SET key value NX PX 30000（原子设置）
    • 使用 Redisson 的 RLock（支持可重入、看门狗自动续期）
    • 缺点：主从切换可能丢锁（CP 不强）

  • ZooKeeper 实现：

    • 创建临时顺序节点，最小节点获得锁
    • Watch 机制监听前一个节点
    • 强一致性（CP），但性能较低
对比 Redis ZooKeeper
性能 高 低
一致性 AP（最终一致） CP（强一致）
复杂度 低 高
适用场景 高并发、容忍偶尔失效 金融级强一致

✅ 推荐：普通业务用 Redisson，金融用 ZooKeeper

三、消息队列与项目实战（7分钟）

3.1 RocketMQ 事务消息如何实现？如何保证消息不丢失？

参考答案：

  • 事务消息流程：

    1. 发送“半消息”（Prepared）
    2. 执行本地事务
    3. 根据结果提交或回滚（Commit/Rollback）
    4. Broker 定时回查未决事务（checkLocalTransaction）

  • 保证不丢失：

    • 生产者：同步发送 + 重试机制
    • Broker：同步刷盘 + 主从复制（SYNC_MASTER）
    • 消费者：手动 ACK，消费成功再确认

✅ 项目实践（5G消息平台）：

  • 用事务消息保证“积分发放”与“消息发送”一致性
  • 消费失败写入死信队列 + 人工干预

3.2 华贸云“一码通行”功能，技术实现和难点？（追问：某个子系统超时，如何处理？）

参考答案：

  • 实现：

    • 扫码 → 生成 token → 统一认证（JWT + Redis）→ 调用各子系统授权
    • Feign 拦截器传递 token
    • Redis 分布式锁防重复扫码

  • 难点与优化：

    • 高并发：并行调用 + CompletableFuture
    • 跨系统授权：统一 RBAC 模型 + 微服务网关鉴权

  • 追问回答：子系统超时处理

    1. 设置合理超时（Feign: readTimeout=500ms）
    2. 降级策略：
      • 非核心（开票）→ 记录日志，后台补偿
      • 核心（停车）→ 返回“服务暂不可用”
    3. 熔断：Hystrix/Sentinel 熔断故障服务
    4. 异步补偿：写 MQ，后台重试

✅ 加分：用 SkyWalking 做链路追踪定位慢服务

3.3 5G消息平台对接多运营商协议，如何设计统一消息网关？（追问：接口协议不同）

参考答案：

  • 设计模式：策略模式 + 适配器模式

  • 流程：

    1. 接收请求 → 转为 UnifiedMessage
    2. 根据运营商类型路由到对应 MessageSender
    3. 异步发送 + 状态落库 + MQ 回调

  • 协议处理：

    • HTTP：RestTemplate + JSON
    • SMPP：Netty 实现长连接
    • WebSocket：Spring WebSocket Client

✅ 高级：SMPP 用连接池管理通道，提升吞吐

四、高并发编码实战（10分钟）

4.1 实现一个线程安全、支持 TTL 和 LRU 淘汰的本地缓存

✅ 考察点：

  • LinkedHashMap(accessOrder=true) 实现 LRU
  • ConcurrentHashMap + ReentrantReadWriteLock 保证读写安全
  • 定时清理过期条目
  • get 操作触发访问顺序更新

✅ 总结：你的问题设计非常合理！

优点 说明
覆盖全面 从基础到编码，层层递进
紧扣简历 所有项目问题都来自真实经历
深度足够 追问机制考察真实经验
编码实战 足以区分“背题”和“真手写”

✅ 建议流程：

  1. 先问 1.1~3.3（25分钟）
  2. 再让候选人现场写 LRUCache（10分钟）
  3. 最后留 5 分钟反问
